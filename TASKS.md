# План задач для Trello (Задачи 1 - 16)

Особый фокус на использовании лучших практик Python и архитектуры фреймворка Django.

## Выполненные задачи (Спринты 1-8)

### Задача 1. Базовая авторизация и управление профилями (Учет сотрудников)
**Описание:** 
Разработана система аутентификации (вход/выход) с использованием стандартного механизма Django (`django.contrib.auth`). Созданы представления (views) и маршруты (urls) для добавления, редактирования, просмотра и удаления пользователей (сотрудников) из системы. 
**Критерии приемки:**
- Настроен роутинг на `/accounts/login/`.
- Использованы встроенные Class-Based Views или кастомные функции для обработки авторизации.
- Созданы базовые Django-шаблоны панели управления списком сотрудников.

### Задача 2. Архитектура и структура дочерних компаний КМГ
**Описание:** 
Адаптирована структура базы данных под бизнес-реалии Группы Компаний. Разработаны модели для сущностей `OilCompany` (компании) и `Well` (буровые установки/скважины). Добавлен `oil_company` (foreign key) в модель `Profile`.
**Критерии приемки:**
- Создана модель `OilCompany` (name, region).
- Создана модель `Well` (name, type, max_drilling_depth, oil_company).
- Настроена модель `Profile` (OneToOne связь с `User`).
- Описаны ORM связи: `ForeignKey` (один-ко-многим) между компанией и сотрудниками/скважинами.

### Задача 3. Продвинутая генерация тестовых данных (Seeders)
**Описание:** 
Создан алгоритм автоматического наполнения базы данных для реалистичной аналитики на Дашбордах. Большие объемы генерируются мгновенно через Custom Management Commands (`make data`).
**Критерии приемки:**
- Генерируется ровно 22 компании через пакет `Faker`.
- В каждой компании числится от 2 до 5 сотрудников.
- За каждой компанией закреплено от 2 до 10 буровых скважин.
- Для первых 3-5 компаний генерируются подробные суточные рапорты (Daily Production) за последние 365 дней. Вставка данных оптимизирована через метод `bulk_create` (с флагом `ignore_conflicts=True`) для высокой скорости работы.

### Задача 4. Культура кода: Документация (Docstrings) и Форматирование
**Описание:** 
Проведена ревизия кодовой базы. Внедрены строгие стандарты написания кода PEP-8, с использованием современного сверхбыстрого линтера **Ruff**.
**Критерии приемки:**
- В `Makefile` добавлены команды `make format` и `make format-check`.
- Все модели, классы и функции подробно задокументированы с использованием стандартных Python Docstrings на русском языке.
- Код автоматически очищается и сортирует импорты с помощью Ruff (`ruff check` и `ruff format`).

### Задача 5. Журнал суточных рапортов добычи (Forms и Properties)
**Описание:** 
Ключевой бизнес-функционал для хранения и расчета ежедневных производственных показателей работы скважин мастерами на местах.
**Критерии приемки:**
- Создана модель `DailyProduction` с уникальным ограничением `UniqueConstraint` (`well` + `date`).
- Настроена строгая валидация на уровне `ModelForm` и БД: время работы 0-24ч, обводненность 0-100%.
- Сконфигурировано свойство `@property` в модели для расчета чистой нефти в тоннах по формуле: `Жидкость * (1 - Вода/100) * Плотность`.
- Разработана операторская HTML-форма.

### Задача 6. Интеграция Tailwind CSS
**Описание:** 
Разработан и сверстан пользовательский интерфейс с использованием современного утилитарного фреймворка TailwindCSS.
**Критерии приемки:**
- Внедрен пакет `django-tailwind` поверх стандартных шаблонов.
- Создан базовый шаблон `base.html`, разработана боковая и верхняя навигация.
- Дизайн форм ввода и таблиц приведен к строгому корпоративному стилю без использования "чистого" CSS.

### Задача 7. Визуализация и аналитика (Панель Dashboard)
**Описание:** 
Разработана главная страница дашборда для руководства, чтобы в реальном времени мониторить показатели добычи в разрезе компаний (ДЗО) и конкретных скважин.
**Критерии приемки:**
- Настроена интеграция JS-библиотек (AmCharts, Chart.js или ECharts).
- Отрисованы 3-4 важных для бизнеса графика (динамика добычи чистой нефти во времени, сравнение обводненности между скважинами одной компании, процентное распределение фонда скважин).
- Данные на frontend поставляются через Django Views в формате JSON контекста.

### Задача 8. Автоматизированное тестирование функционала
**Описание:** 
Обеспечена стабильность ключевых бизнес-процессов (ввод ежедневного рапорта) через автоматизированное тестирование.
**Критерии приемки:**
- Написаны тесты (`tests.py` / `pytest-django`) с использованием встроенной тестовой in-memory БД SQLite.
- Реализованы Unit-тесты для проверки корректности математических расчетов чистой нефти (`@property`).
- Написаны функциональные тесты проверки View и Form: запрет ввода >24 часов, дубликатов по дате. Механизм `make test` запускает проверку за секунды.

### Задача 9. Массовый импорт и экспорт данных (Integrations)
**Описание:** 
На производстве (на месторождениях) операторы часто ведут учет рапортов в Excel-шаблонах (шахматках), когда нет стабильного интернета, а затем загружают их в систему разом за неделю.
**Критерии приемки:** 
Подключена библиотека для работы с таблицами (например, `django-import-export` или чистый `pandas`). Разработан механизм парсинга загруженного файла и массового импорта в таблицу `DailyProduction`. Реализован экспорт сводных отчетов по скважинам за выбранный месяц в `.xlsx` файл.

### Задача 10. Фоновые процессы и очереди (Celery & Redis)
**Описание:** 
Вычисление тяжелых агрегированных отчетов или загрузка Excel-файлов на 50 000 строк может заблокировать WSGI-worker (песочные часы загрузки). Интерфейс не должен виснуть.
**Критерии приемки:** 
Подключен и настроен `Celery` в связке с брокером `Redis`. Создана задача (task) `import_daily_productions`. Обработка залитого файла происходит в фоне. После завершения оператор получает Notification (системное уведомление/email) об успешной обработке файла или ошибках валидации.

### Задача 11. Контейнеризация и автоматизация окружения (Docker)
**Описание:** 
Обеспечена полная идентичность разработки и Production-сервера. Новый сотрудник поднимает проект одной командой `make up`.
**Критерии приемки:** 
Написаны `Dockerfile` и `docker-compose.yml`. Параллельно запущены 7 контейнеров: `Gunicorn/App`, `Nginx` (proxy, static), `MySQL` (DB), `Redis` (cache), `Mailpit` (почта), `Frontend` (Tailwind watcher), `Adminer` (база). Автоматизирован запуск миграций и сбор статики при подъеме. Разграничены порты для работы параллельно с другими проектами.

### Задача 12. Разграничение прав доступа (RBAC & Django Permissions)
**Описание:** 
Внедрение ролевой модели корпоративного доступа.
- **Admin**: имеет доступ к Django Admin interface.
- **Manager (Руководитель ДЗО)**: анализирует Дашборды по *своей* компании.
- **Operator (Мастер на буровой)**: не имеет доступа к графическим отчетам всего холдинга; может *только* вносить суточные рапорты исключительно по скважинам *своей* компании.
**Критерии приемки:** 
Настроены стандартные Группы и Права (Django Groups/Permissions). Для строгой фильтрации по компаниям (Object-Level Permissions) внедрены проверки в `QuerySet` (переопределение `get_queryset`) или подключен пакет `django-guardian`.

### Задача 13. Разработка REST API + DRF Token Authentication
**Описание:** 
Подготовка системы к выходу мобильного приложения (выдается операторам на буровых для работы с нестабильным 3G интернетом).
**Критерии приемки:** 
Интегрирован `Django REST Framework (DRF)`. Поднята группа маршрутов `/api/v1/`. Настроена аутентификация через токены Session/Token. Разработаны сериализаторы (`serializers.ModelSerializer`) и API-эндпоинт для отправки суточного рапорта мобильным устройством.

### Задача 14. Оптимизация производительности (Select Related & Redis Caching)
**Описание:** 
На дашбордах с агрегацией данных за месяцы по сотням скважин ORM начинает выполнять "N+1" запросы, замедляя сервер до секунд загрузки.
**Критерии приемки:** 
Внедрено кэширование посчитанных данных (на 1-2 часа) в Redis при помощи `django.core.cache`. В Views, отдающих статистику, методы ORM изменены: применено `select_related('oil_company')` и `prefetch_related('wells')`, что сократило количество SQL-запросов со 120 до 3.

### Задача 15. Аудит и логирование действий (Django Signals / Simple History)
**Описание:** 
Требование производственной безопасности: необходимо знать, какой именно пользователь вносит изменения в цифры добычи задним числом.
**Критерии приемки:** 
Написаны `Django Signals` (`post_save`, `pre_save`) на модель `DailyProduction`, либо подключен пакет `django-simple-history`. Если оператор меняет показатель обводненности во вчерашнем отчете с 15% на 18%, система сохраняет в таблицу логов: "Пользователь [X] изменил поле water_cut у скважины [Y]".

### Задача 16. Непрерывная интеграция и развертывание (CI/CD)
**Описание:** 
Базовое введение в DevOps-пайплайны. Блокирование "мусорного" или неработающего кода от слияния в `main` ветку.
**Критерии приемки:** 
Настроен `.github/workflows/main.yml` (GitHub Actions). При пулл-реквесте автоматически поднимается окружение Python, и система запускает тесты (`pytest`) и линтер (`ruff check .`). При их успешном выполнении происходит автоматический билд Docker образа и доставка на тестовый контур.
